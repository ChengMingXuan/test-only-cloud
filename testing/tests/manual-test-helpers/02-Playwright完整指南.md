
1. 启动浏览器窗口
2. 自动推进到第一个测试步骤
3. 给你一个**用户交互提示**
4. 等待你的确认（✅ 按钮或 ❌ 标记问题）
5. 继续下一步
6. 最后生成报告

预计耗时：**5-10 分钟**

## 📚 完整使用指南

### 基本概念

#### 什么是"半自动化"测试？

```
完全手工测试：
✅ 效果好（人工判断准确）
❌ 时间长（每个功能 30+ 分钟）
❌ 易出错（重复性任务容易遗漏）

完全自动化测试：
✅ 速度快（几分钟完成全部）
❌ 维护难（脚本更新成本高）
❌ 不灵活（无法适应快速变化）

🎯 半自动化测试（我们的方案）：
✅ 自动化 70%（框架、导航、数据准备）
✅ 人工确认 30%（关键决策、UI 验证）
✅ 平衡效率和准确性
```

#### 一个典型的测试步骤

```
步骤 A：设置和准备（自动）
  └─ 系统登录、打叫相关页面、注入测试数据
  └─ 耗时：2-3 秒

步骤 B：用户操作（人工）
  └─ 系统提示："请创建一个新工单，包含以下信息..."
  └─ 你在浏览器中操作（填表、提交等）
  └─ 耗时：1-3 分钟

步骤 C：结果验证（自动 + 人工）
  └─ 系统自动检查返回的 API 响应（自动）
  └─ 系统提示你观察页面变化（人工）
  └─ 你确认："✅ 工单已成功创建" 或 "❌ 出现了某某错误"
  └─ 耗时：1-2 分钟

步骤 D：报告记录（自动）
  └─ 系统自动截图、记录 API、保存日志
  └─ 耗时：< 1 秒

总计：约 5-10 分钟/步
```

### 故障排查

#### 问题 1：浏览器启动失败

```
错误信息：
  Error: browserType.launch() timed out after 30000ms

可能原因：
  • Playwright 浏览器未安装
  • 系统内存不足
  • 防火墙阻止

解决方法：
  # 重新安装浏览器
  npx playwright install

  # 检查磁盘空间（需要 > 500MB）
  df -h

  # 重新启动 Docker 中的测试系统
  docker-compose restart
```

#### 问题 2：连接超时

```
错误信息：
  Error: net::ERR_CONNECTION_REFUSED

可能原因：
  • 测试系统未启动
  • 系统地址配置错误
  • 网络连接问题

解决方法：
  # 检查系统是否运行
  curl http://localhost:5000/

  # 启动测试系统
  docker-compose up -d

  # 检查网络连接
  ping localhost
```

#### 问题 3：测试步骤超时

```
错误信息：
  Timeout of 5 minutes exceeded while waiting for user confirmation

可能原因：
  • 网络延迟很高（某个 API 响应慢）
  • 页面加载时间长
  • 浏览器卡顿

解决方法：
  # 查看网络延迟
  # （在报告截图中查看"Network"选项卡）

  # 重启浏览器
  npx playwright install

  # 逐个运行场景，找出最慢的
  npm run scenario:01 -- --debug
```

#### 问题 4：取消测试

```
如果测试进行中想要停止：
  • 按 Ctrl+C 停止当前测试
  • 所有已完成的步骤仍会被记录到报告中
  • 可以稍后继续运行同一场景

提示：
  • 不推荐中途停止（可能导致数据不一致）
  • 如果必须停止，请手动清理测试数据
  • 查看 scripts/cleanup-test-data.sh 获取帮助
```

### 最佳实践

#### 运行场景时的黄金法则

1. **选择正确的时间**
   - ✅ 系统空闲时运行（避免高峰期）
   - ✅ 网络良好时运行（避免网络拥塞）
   - ❌ 其他团队在部署时不要运行

2. **做好准备**
   - ✅ 预留足够的时间（至少预留比预计时间 +50%）
   - ✅ 确保测试数据库是最新状态
   - ✅ 检查系统是否正常运行
   - ✅ 有笔和纸记录问题（或打开记事本）

3. **关注过程**
   - ✅ 不要走神（半自动化需要人工观察）
   - ✅ 仔细阅读每个步骤的提示
   - ✅ 如有不确定，给出 ❌（宁愿多报告问题）
   - ✅ 记录任何异常和意图情况

4. **分析结果**
   - ✅ 第一时间查看失败的步骤
   - ✅ 查看截图对比预期的行为
   - ✅ 阅读错误日志和 API 响应
   - ✅ 向开发人员报告问题时附带报告

#### 避免常见错误

| 错误 | 后果 | 解决方案 |
|------|------|---------|
| 不观察页面，直接点 ✅ | 遗漏 bug | 耐心观察，给出准确反馈 |
| 中途停止测试 | 数据混乱 | 完整运行或手动清理数据 |
| 同时运行多个场景 | 测试干扰 | 一个一个运行 |
| 在系统高峰期运行 | 超时失败 | 选择空闲时间 |
| 忽视报告 | 遗留问题 | 认真阅读和分析报告 |
