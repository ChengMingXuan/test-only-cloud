# AIOPS 平台五工具互补测试架构方案

> **版本**: v1.0  
> **日期**: 2026年3月5日  
> **作者**: JGSY.AGI 测试团队  
> **目标**: 通过 Playwright + Selenium + Puppeteer + Cypress + K6 构建立体化、全覆盖的测试体系

---

## 一、架构设计理念

### 1.1 设计原则

✅ **互补不重复**: 每个工具聚焦特定场景，避免功能重叠  
✅ **分层不割裂**: 从单元→组件→页面→流程→系统形成完整链条  
✅ **真实不虚拟**: 优先使用真实浏览器，确保生产环境一致性  
✅ **快速不失真**: 快速测试用于CI，完整测试用于发布门禁  

### 1.2 五工具定位矩阵

| 工具 | 核心定位 | 优势场景 | 覆盖率目标 | 执行频率 |
|------|---------|---------|-----------|---------|
| **Playwright** | 主力E2E测试 | 跨浏览器核心业务流程 | 60-70% 关键路径 | 每次构建 |
| **Selenium** | 兼容性测试 | 多浏览器矩阵、老旧浏览器 | 20-30% 兼容性场景 | 每日/发版前 |
| **Puppeteer** | 性能监控 | Chrome深度集成、PDF/截图 | 10-15% 性能关键页面 | 每日基准 |
| **Cypress** | 组件交互测试 | 单页应用、前端开发调试 | 40-50% 前端组件 | 开发阶段 |
| **K6** | 性能压测 | API负载、并发、基准测试 | 100% API端点 | 每周/发版前 |

### 1.3 测试金字塔模型

```
        ┌─────────────┐
        │  K6 性能测试  │  ← 系统级负载/压力/基准测试
        └─────────────┘
       ┌───────────────┐
       │ Playwright E2E │  ← 跨浏览器端到端业务流程
       └───────────────┘
      ┌─────────────────┐
      │ Selenium 兼容性  │  ← 浏览器矩阵兼容性验证
      └─────────────────┘
     ┌───────────────────┐
     │ Puppeteer 性能监控 │  ← Chrome性能剖析/视觉回归
     └───────────────────┘
    ┌─────────────────────┐
    │  Cypress 组件交互    │  ← 前端组件、交互、状态测试
    └─────────────────────┘
   ┌───────────────────────┐
   │   pytest API 单元测试  │  ← API接口、数据库、业务逻辑（已有）
   └───────────────────────┘
```

## 二、工具职责详细分工

### 2.1 Playwright（主力E2E测试 - 60-70%覆盖）

**核心职责**:
- ✅ 跨浏览器（Chromium/Firefox/WebKit）端到端测试
- ✅ 关键业务流程自动化（登录→操作→结果验证）
- ✅ 多页面协同、页面跳转、表单提交
- ✅ 断言UI状态、数据库状态、API响应

**典型场景**:
```javascript
// 示例：充电订单全流程
test('充电订单从创建到支付完成', async ({ page }) => {
  await page.goto('/charging/orders/create');
  await page.fill('#stationId', '站点001');
  await page.fill('#powerKwh', '30');
  await page.click('button:has-text("创建订单")');
  await expect(page.locator('.order-id')).toBeVisible();
  // 跨页面跳转到支付
  await page.click('text=立即支付');
  await page.selectOption('#paymentMethod', 'wechat');
  await page.click('button:has-text("确认支付")');
  await expect(page.locator('.success-message')).toContainText('支付成功');
});
```

**浏览器矩阵**:
- Chrome (Chromium) - 主力
- Firefox - 次要
- Safari (WebKit) - Mac/iOS用户

**执行时机**: 每次 Git Push → CI Pipeline

### 2.2 Selenium（兼容性测试 - 20-30%覆盖）

**核心职责**:
- ✅ 多浏览器兼容性矩阵验证
- ✅ 老旧浏览器支持（IE11, Edge Legacy）
- ✅ Selenium Grid 分布式执行
- ✅ 移动浏览器模拟（Android Chrome/iOS Safari）

**典型场景**:
```python
# 示例：登录页在不同浏览器的兼容性
@pytest.mark.parametrize("browser", ["chrome", "firefox", "edge", "safari"])
def test_login_cross_browser(browser):
    driver = get_driver(browser)
    driver.get("https://aiops.jgsy.com/login")
    
    # 验证关键元素在所有浏览器中正确渲染
    assert driver.find_element(By.ID, "username").is_displayed()
    assert driver.find_element(By.ID, "password").is_displayed()
    assert driver.find_element(By.CSS_SELECTOR, "button[type='submit']").is_enabled()
    
    # 验证CSS兼容性（如flex布局）
    login_form = driver.find_element(By.CLASS_NAME, "login-form")
    assert login_form.value_of_css_property("display") == "flex"
```

**浏览器矩阵**:
- Chrome 120+ / 110-119 / 90-109
- Firefox Latest / ESR
- Edge Latest / Legacy
- Safari 16+ / 15
- Mobile: Android Chrome, iOS Safari

**执行时机**: 每日定时任务 + 发版前完整回归

### 2.3 Puppeteer（性能监控 - 10-15%覆盖）

**核心职责**:
- ✅ Chrome DevTools Protocol 深度集成
- ✅ 页面性能指标采集（FCP/LCP/TTI/CLS）
- ✅ 网络请求拦截和性能分析
- ✅ PDF生成、截图、视觉回归测试

**典型场景**:
```javascript
// 示例：首页性能基准测试
const puppeteer = require('puppeteer');

(async () => {
  const browser = await puppeteer.launch();
  const page = await browser.newPage();
  
  // 启用性能监控
  await page.tracing.start({ path: 'trace.json', screenshots: true });
  await page.goto('https://aiops.jgsy.com/dashboard', { waitUntil: 'networkidle2' });
  await page.tracing.stop();
  
  // 采集核心Web性能指标
  const metrics = await page.metrics();
  const performanceTiming = JSON.parse(await page.evaluate(() => 
    JSON.stringify(window.performance.timing)
  ));
  
  // 断言性能SLA
  const domContentLoaded = performanceTiming.domContentLoadedEventEnd - performanceTiming.navigationStart;
  const loadComplete = performanceTiming.loadEventEnd - performanceTiming.navigationStart;
  
  console.log(`DOM Content Loaded: ${domContentLoaded}ms (SLA: <2000ms)`);
  console.log(`Page Load Complete: ${loadComplete}ms (SLA: <5000ms)`);
  
  assert(domContentLoaded < 2000, 'DOMContentLoaded超过2秒');
  assert(loadComplete < 5000, '页面完整加载超过5秒');
  
  // 生成性能报告PDF
  await page.pdf({ path: 'performance-report.pdf', format: 'A4' });
  
  await browser.close();
})();
```

**监控指标**:
- FCP (First Contentful Paint) < 1.8s
- LCP (Largest Contentful Paint) < 2.5s
- TTI (Time to Interactive) < 3.8s
- CLS (Cumulative Layout Shift) < 0.1
- TBT (Total Blocking Time) < 300ms

**执行时机**: 每日定时基准测试 + 性能回归检测

### 2.4 Cypress（组件交互测试 - 40-50%覆盖）

**核心职责**:
- ✅ 单页应用（SPA）组件测试
- ✅ 前端交互、状态管理、路由跳转
- ✅ 实时重载、时间旅行调试
- ✅ 网络请求 Mock 和 Stub

**典型场景**:
```javascript
// 示例：充电站列表筛选和分页
describe('充电站管理页面', () => {
  beforeEach(() => {
    cy.login('admin@jgsy.com', 'Admin@123'); // 自定义命令
    cy.visit('/stations');
  });
  
  it('按城市筛选充电站', () => {
    cy.get('[data-testid="city-filter"]').click();
    cy.get('[data-value="beijing"]').click();
    cy.get('.station-list').should('have.length.at.least', 1);
    cy.get('.station-list').each(($el) => {
      cy.wrap($el).find('.city').should('contain', '北京');
    });
  });
  
  it('分页加载更多充电站', () => {
    cy.get('.station-list').should('have.length', 10);
    cy.get('[data-testid="load-more"]').click();
    cy.get('.station-list').should('have.length', 20);
  });
  
  it('搜索充电站（模拟API响应）', () => {
    cy.intercept('GET', '/api/stations/search?keyword=*', {
      statusCode: 200,
      body: { data: [{ id: 1, name: '测试站点', city: '北京' }] }
    }).as('searchStations');
    
    cy.get('[data-testid="search-input"]').type('测试站点');
    cy.wait('@searchStations');
    cy.get('.station-list').should('have.length', 1);
    cy.get('.station-list').first().should('contain', '测试站点');
  });
});
```

**测试重点**:
- React 组件生命周期
- Redux 状态管理
- 表单验证和错误提示
- 模态框、下拉菜单、日期选择器
- WebSocket 实时更新

**执行时机**: 前端开发实时运行 + CI Pipeline

### 2.5 K6（性能压测 - 100% API覆盖）

**核心职责**:
- ✅ API端点性能基准测试
- ✅ 负载测试（模拟N个并发用户）
- ✅ 压力测试（找到系统瓶颈）
- ✅ 浸泡测试（长时间稳定性）

**典型场景**:
```javascript
// 示例：充电订单创建API压力测试
import http from 'k6/http';
import { check, sleep } from 'k6';

export let options = {
  stages: [
    { duration: '2m', target: 100 },  // 2分钟爬升到100用户
    { duration: '5m', target: 100 },  // 维持5分钟
    { duration: '2m', target: 200 },  // 再爬升到200用户
    { duration: '5m', target: 200 },  // 维持5分钟
    { duration: '2m', target: 0 },    // 2分钟降到0
  ],
  thresholds: {
    'http_req_duration': ['p(95)<500', 'p(99)<1000'], // 95%请求<500ms, 99%<1s
    'http_req_failed': ['rate<0.01'],                 // 错误率<1%
  },
};

export default function () {
  const url = 'https://aiops.jgsy.com/api/charging/orders';
  const payload = JSON.stringify({
    stationId: 'station-001',
    userId: 'user-123',
    powerKwh: 30,
    paymentMethod: 'wechat',
  });
  
  const params = {
    headers: {
      'Content-Type': 'application/json',
      'Authorization': `Bearer ${__ENV.AUTH_TOKEN}`,
    },
  };
  
  const res = http.post(url, payload, params);
  
  check(res, {
    '状态码200': (r) => r.status === 200,
    '返回订单ID': (r) => JSON.parse(r.body).data.orderId !== undefined,
    '响应时间<500ms': (r) => r.timings.duration < 500,
  });
  
  sleep(1); // 模拟用户思考时间
}
```

**测试类型**:
- 冒烟测试: 1-10 VU, 1分钟
- 负载测试: 100-500 VU, 10分钟
- 压力测试: 500-2000 VU, 找到崩溃点
- 浸泡测试: 50 VU, 8小时

**执行时机**: 每周性能基准 + 发版前压测

## 四、统一测试执行流程

### 4.1 本地开发阶段

```bash
# 开发者在本地运行快速测试套件
npm run test:quick

# 内部执行顺序：
# 1. pytest 单元测试（5分钟）
# 2. Cypress 组件测试（10分钟）
# 3. Playwright 冒烟测试（5分钟）
# 总计：20分钟
```

### 4.2 CI Pipeline（每次Push）

```yaml
# .github/workflows/test.yml
name: 全量测试流水线

on: [push, pull_request]

jobs:
  unit-tests:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Pytest API单元测试
        run: |
          pip install -r tests/test-automation/requirements.txt
          pytest tests/test-automation/ -v --tb=short
        timeout: 10m
  
  component-tests:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Cypress 组件测试
        run: |
          npm install
          npm run cypress:run:component
        timeout: 15m
  
  e2e-tests:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        browser: [chromium, firefox, webkit]
    steps:
      - uses: actions/checkout@v3
      - name: Playwright E2E测试
        run: |
          npm install
          npx playwright test --project=${{ matrix.browser }}
        timeout: 30m
  
  performance-tests:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Puppeteer 性能基准
        run: |
          npm install
          node tests/puppeteer-tests/performance-baseline.js
        timeout: 10m
```

## 九、常见问题

**Q1: 为什么需要5个工具？一个Playwright不够吗？**  
A: 每个工具都有独特优势。Playwright适合E2E，但不如Cypress对SPA友好；Selenium支持最广泛的浏览器矩阵；Puppeteer提供最深度的Chrome性能剖析；K6是专业的性能压测工具。组合使用才能覆盖所有场景。

**Q2: 会不会重复测试导致浪费时间？**  
A: 不会。我们明确了每个工具的职责边界，避免重复。例如：登录功能，Playwright测试跨浏览器流程，Cypress测试登录组件交互，Selenium验证5年前的旧浏览器，Puppeteer监控登录页性能，K6压测登录API并发。

**Q3: 维护成本会不会太高？**  
A: 通过统一的Page Object模式、共享的测试数据工厂、中央化的配置管理，大幅降低维护成本。且测试金字塔底层（pytest/Cypress）执行频率高但变更少，顶层（Selenium/K6）执行频率低但覆盖广。

**Q4: 如何决定某个功能用哪个工具测？**  
A: 参考工具定位矩阵（第一节）。一般原则：
- 新功能开发阶段 → Cypress快速反馈
- 功能稳定后 → Playwright E2E覆盖
- 发版前 → Selenium兼容性检查
- 性能关键页面 → Puppeteer持续监控
- API端点 → K6性能基准

**Q5: 测试环境如何隔离？**  
A: 每个工具使用独立的测试租户（tenant_id），数据库通过软删除标记隔离，测试执行前通过 `cleanup-test-data.ps1` 清理。详见测试数据管理规范。
